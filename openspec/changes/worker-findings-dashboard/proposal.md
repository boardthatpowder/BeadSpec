## Why

Ruflo background workers (`security-audit`, `test-gap-detector`, `cost-tracker`) file Beads issues automatically via `~/.claude/ruflo/hooks/on-finding.sh`, prefixing each note with `Auto-filed by ruflo-<worker> on <ts>. Branch: <b>` and storing a paired memory entry. Today those issues appear in the task list as ordinary `bd` issues with no provenance signal — the user cannot tell at a glance that an issue was auto-generated by a worker, which worker filed it, or how many findings each worker has produced. The follow-up triage skill (`openspec-beads-followup`) already matches these notes by string, so the UI can use the same canonical pattern.

## What Changes

- `TaskListItem` SHALL render a provenance chip when the leading line of the issue's notes matches `^Auto-filed by ruflo-(?<worker>[a-z0-9-]+) on `. The chip text is the worker name (e.g. `security-audit`) and the chip uses the existing `worker:` palette in `LABEL_CHIP_COLORS`.
- `BdHealthPanel` SHALL introduce a tab strip with two tabs: `Checks` (the existing five `bd` checks, unchanged behaviour) and `Worker findings` (new). Default tab is `Checks` to preserve existing entry behaviour.
- The `Worker findings` tab SHALL list every open Beads issue whose notes match the `Auto-filed by ruflo-` pattern, grouped by worker name, with a per-group header showing the worker name, total count, and a severity breakdown derived from `priority` (P1 → critical, P2 → high, P3 → medium, P4 → low). Clicking a finding row SHALL open that issue in `TaskDetailPanel` using the existing nav contract.
- A new Tauri command `list_worker_findings(project_path)` SHALL return an array of `WorkerFinding { issue_id, title, worker, priority, status, notes_first_line, created_at }` by querying Beads issues whose `notes` LIKE `Auto-filed by ruflo-%` and parsing the worker token from the first line.
- The chip on `TaskListItem` SHALL coexist with existing label chips — it is rendered alongside them, not in place of them, and it is inserted *before* the label chip list so provenance is the leftmost signal.

Non-goals (explicit):

- No memory IPC for v1. The paired Ruflo memory entry is not surfaced — only the bd notes pattern.
- No worker invocation from the UI (the `/audit`, `/ruflo-loop testgaps`, `/ruflo-cost` slash commands stay in the agent surface).
- No write path — the dashboard cannot dismiss, snooze, retag, or close findings. Use existing task detail actions.
- No filter dimension auto-derived from the provenance chip. The chip is informational only.
- No backfill or migration. Existing issues filed before this change still match the same notes regex.
- No notification badge on the Health view nav entry. Out of scope for v1.

## Capabilities

### New Capabilities

- `worker-findings`: surfaces Ruflo background-worker provenance from Beads notes both inline on each task list row and as a dedicated grouped view inside the Health panel.

### Modified Capabilities

- `task-list`: adds a requirement for a provenance chip on `TaskListItem` when the issue's notes match the worker-finding pattern.
- `bd-health-panel`: adds a requirement for a tab strip and the `Worker findings` tab; existing `Checks` behaviour is unchanged.

## Impact

- **Tauri command** (new): `list_worker_findings` in `src-tauri/src/commands/workers.rs` (new file). Uses the same `ProjectRegistry` SQL pool as `get_change_dependencies` / `get_change_beads_progress`.
- **Type bindings**: new `WorkerFinding` and `WorkerFindingsSeverity` types auto-generated via `specta` into `src/bindings.ts`.
- **Frontend IPC**: new `listWorkerFindings(projectPath)` wrapper in `src/ipc.ts`.
- **React**: new `src/components/health/WorkerFindingsPanel.tsx`. `BdHealthPanel.tsx` gains a tab strip. `TaskListItem.tsx` gains the provenance chip + a helper `parseWorkerProvenance(notes)`.
- **Data layer**: `Task` already exposes `notes` from the Beads schema — no schema change. If the loaded `Task` type does not currently carry `notes`, the chip parser falls back to a dedicated `list_worker_findings` payload that *does* carry the parsed worker — see Decision 4.
- **No CLI shellouts** — query goes against the Dolt SQL pool.
- **No Ruflo / memory IPC**.
