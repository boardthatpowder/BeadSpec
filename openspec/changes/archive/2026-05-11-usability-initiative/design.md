## Context

beads-ui is a Tauri 2 desktop app reading issue data directly from Dolt SQL (per-project connection pool) and writing via the `bd` CLI. The sync layer polls `dolt_log()` every 2s and pushes changed task IDs into TanStack Query cache via Tauri events. IPC types are code-generated by `tauri-specta`. The user's full workflow stack — `bd`, OpenSpec, ruflo memory, git — already exists in the terminal; beads-ui currently mirrors only the Dolt issue data.

This design covers the **architectural patterns** shared across the 13 sub-changes in this initiative. Each sub-change ships as its own OpenSpec change; this document establishes the contracts and decisions they must follow so they compose cleanly.

## Goals / Non-Goals

**Goals:**
- Define the Tauri command patterns for filesystem reads (OpenSpec), CLI delegation (`bd`, `ruflo`), and git/Dolt history so each sub-change follows a consistent contract
- Establish the in-memory grouping architecture so downstream sub-changes that rely on it (workspace-context, openspec epic rollup) have a stable foundation
- Document the multi-project path resolution approach (project root vs. beads-ui source root) — a footgun that every sub-change must handle correctly
- Identify cross-cutting concerns: quick-capture window lifecycle, task-detail section ordering, KPI bar extension points

**Non-Goals:**
- GitNexus integration (explicitly out of scope)
- "Ask Claude" sidekick (deferred)
- Kanban / board view
- Implementing any sub-change — each has its own design artifact

## Decisions

### D1: Sub-change implementation strategy

Each of the 13 capabilities is implemented as a standalone OpenSpec change (`openspec-new-change <slug>`) that imports into its own beads epic. This document's role is to set shared architecture; it does not ship code itself.

Ordering enforces dependencies:
- `configurable-list-groupings` ships first — `workspace-tag-auto-scope` and the openspec epic-rollup UX both depend on it
- `openspec-spec-panel` can ship before `multi-tab-task-detail-workspace` lands (as a collapsible section); it migrates to a dedicated tab afterward
- `bd-health-panel` and `bd-formulas` can be built in parallel — they share the `run_bd_command` Tauri command but don't share UI surface

### D2: Project-root path resolution

**Problem**: `openspec/changes/`, `.git/`, and the user's project files live in the **project root** being tracked, not the beads-ui source directory. Every new Tauri command that reads these must receive `project_root: PathBuf` (already held per-project in `AppState`).

**Pattern**: All new commands in the `openspec.rs` and `git.rs` modules accept no path argument — they resolve the project root from `AppState::current_project()` internally. This avoids path injection and keeps the TypeScript call sites clean.

**Alternative considered**: Pass `project_root` as a TS argument — rejected because every caller would need to read it from `useProject()` hook and the types would leak an internal concern.

### D3: In-memory grouping (not SQL)

The task list loads all (filtered) tasks via TanStack Query. Grouping is a pure JS transform:

```
groupTasks(tasks: Task[], groupBy: GroupConfig) → GroupedList
```

where `GroupedList` is `Array<{ key: string; label: string; tasks: Task[] }>`.

`GroupConfig`:
- `{ field: 'status' | 'priority' | 'assignee' | 'task_type' }` — group by scalar field
- `{ labelPrefix: string }` — group by label suffix after the first colon
- `null` — no grouping (current behavior)

Persisted in `HashStateContext` as `groupBy` string (`"field:status"`, `"label:openspec"`, etc.). Collapses state persisted in Tauri store (`layout.json`).

**Alternative considered**: Push GROUP BY to SQL — rejected because label-prefix grouping is not expressible in SQL without a SUBSTRING, pagination would break the current task-id-based cache invalidation, and the full dataset is already in memory.

### D4: Tauri command modules for new integrations

Two new Rust command modules, registered in `lib.rs`:

**`src-tauri/src/commands/openspec.rs`**
- `list_changes(project_root) → Vec<ChangeInfo>` — reads `openspec/changes/` dirs; each entry has `name`, `status`, `progress`, `is_archived`
- `read_change_artifact(change, artifact) → String` — reads a single file (e.g. `proposal.md`, `specs/list-grouping/spec.md`)
- `get_change_progress(change) → ChangeProgress` — parses `tasks.md` checkbox lines (`- [x]` / `- [ ]`)
- `run_openspec_validate(change) → ValidationResult` — shells `openspec validate --change <name> --json`
- `import_change_to_beads(change) → CommandOutput` — shells `openspec-beads-import` (non-streaming, returns stdout)

**`src-tauri/src/commands/external.rs`**
- `run_bd_command(args: Vec<String>) → CommandOutput` — runs `bd <args>` in project root, captures stdout/stderr, structured result. Used by health-panel, formulas, human-queue.
- `run_ruflo_command(args: Vec<String>) → CommandOutput` — runs `ruflo <args>`, same pattern. Used by memory-panel.
- `get_git_refs_for_issue(issue_id: String) → GitRefs` — runs `git log --oneline --all --grep=<id>` + `git branch --list` in project root; returns `{ commits: Vec<CommitRef>, branches: Vec<String> }`
- `get_dolt_history_for_issue(issue_id: String) → Vec<DoltRevision>` — queries `dolt_log` filtered to rows affecting this task id; returns field diffs

`CommandOutput` = `{ stdout: String, stderr: String, exit_code: i32 }`. All commands time out at 10s (configurable). `run_bd_command` / `run_ruflo_command` are **read-only** guards — commands that write (e.g. `bd mol pour`) go through a dedicated confirmation flow in the UI before the Tauri call.

### D5: Workspace context detection

`get_workspace_context() → WorkspaceContext` (in `external.rs`) runs `git rev-parse --abbrev-ref HEAD` in the project root and returns `{ branch: String, label_branch: String, label_worktree: String, label_repo: String }`. Label values are derived from the path the same way `ruflo_key_prefix` does (branch = `branch:<HEAD>`, worktree = last path segment, repo = git remote-stripped or dir name).

Falls back to `{ branch: "unknown", … }` if the project root is not a git repo. Result is cached in `AppState` and re-fetched on project switch.

### D6: Quick-capture window

A second Tauri window (`quick-capture`) registered in `tauri.conf.json` with `visible: false`. The global shortcut `CmdOrCtrl+Shift+N` (registered via `tauri-plugin-global-shortcut`) toggles it. On show, the window fetches `get_workspace_context()` and pre-populates the three label chips. On submit it calls the existing `create_task` Tauri command and closes.

No new React router route — the window has its own `quick-capture.html` entry point with a minimal React tree (no layout shell).

**Alternative considered**: Tray popover — rejected because the tray popover already exists (`TrayPopover.tsx`) and is scoped to tray interactions; mixing quick-capture into it would muddle its purpose.

### D7: Task detail section ordering and `multi-tab-task-detail-workspace` coexistence

The three additive task-detail sections (OpenSpec panel, Ruflo memory panel, Git/Dolt history) are implemented as collapsible accordion sections appended below the existing Activity tab content. When `multi-tab-task-detail-workspace` lands and restructures the detail pane into tabs, the OpenSpec panel migrates to a dedicated "Spec" tab; the other two move into "History" and "Context" tabs respectively.

Each section is a self-contained component that renders `null` when its data source is unavailable (no `openspec:` label, no ruflo results, no git history). No layout shift when absent.

### D8: KPI bar extension points

The KPI bar (`KpiBar.tsx`) is extended with a `mode` prop: `'counts'` (current) | `'burndown'` | `'velocity'`. A dropdown on the bar switches modes. The velocity/burndown data is derived from `task_history` rows (already fetched per issue) aggregated across issues — no new Tauri command needed. A `useBurndown(changeLabel: string)` hook does the aggregation client-side.

### D9: Charting library for burndown

`recharts` — lightweight, tree-shakeable, already used elsewhere in the workspace. If not yet installed in beads-ui, `bun add recharts`. No alternative needed; `recharts` is the canonical choice here.

## Risks / Trade-offs

- **Project-root confusion** → Any sub-change that forgets to route filesystem/git calls through `AppState::current_project()` will silently read the wrong directory. Mitigation: the `external.rs` and `openspec.rs` modules are the **only** place that touches the filesystem outside of the DB layer — PR review checks for any stray `std::fs::` calls in other modules.

- **`bd` / `ruflo` CLI not on PATH in Tauri sandbox** → macOS Tauri apps do not inherit the user's shell `PATH`. `run_bd_command` must resolve the binary path at startup via `which bd` (shelled before sandboxing) and cache the result. Mitigation: add `bd_path` and `ruflo_path` to `AppState`; fail gracefully with a "CLI not found" message in the UI.

- **OpenSpec `validate` is slow** → `openspec validate` can take 1–3s. Mitigation: run it lazily (on demand via button, not on panel open) and cache the last result with a staleness indicator.

- **Quick-capture shortcut conflicts** → `CmdOrCtrl+Shift+N` may conflict with other apps. Mitigation: make the shortcut user-configurable via the existing shortcuts modal; document the default.

- **`multi-tab-task-detail-workspace` migration** → The three additive sections described in D7 will need a second commit when the workspace change lands. Mitigation: each section component accepts a `containerMode: 'section' | 'tab'` prop from day one so the migration is a one-liner prop change.

## Migration Plan

All changes are additive — no existing data model changes, no existing Tauri command changes, no store schema migrations. Sub-changes ship independently. Feature-flag approach not needed; each capability is opt-in by nature (grouping defaults to off, panels render null when inapplicable, quick-capture shortcut is globally registered but silent until triggered).

## Open Questions

- **`recharts` install**: confirm it is not already in `package.json` before the velocity-burndown sub-change adds it.
- **`openspec-beads-import` path**: the import action shells the `openspec-beads-import` skill — verify the skill is accessible as a CLI command from the Tauri process PATH, or document the fallback (copy the shell command to clipboard for the user to run).
- **Ruflo memory query format**: confirm `ruflo memory search <query> --json` returns a stable JSON schema before the ruflo-memory-panel sub-change specifies against it.
- **`dolt_diff_tasks` availability**: verify the Dolt diff table is exposed in the current schema before the git-history-panel sub-change depends on it; may need a raw `dolt_log()` + diff query instead.
