## Context

beads-ui is a Tauri 2 desktop app (Rust + React/TypeScript + Tailwind v4). The app already watches `.beads/issues.jsonl` for live task updates via `JsonlWatcher` in `src-tauri/src/db/watchers.rs`. IPC types are auto-generated by `tauri-specta`. Project root is resolved from `AppState::current_project()` — sub-changes must not accept path arguments from TypeScript (per usability-initiative design D2).

The `openspec-change-browser` sub-change adds one new top-level view and the Tauri backend plumbing to power it. It follows the exact same watcher pattern established by `JsonlWatcher` to watch `openspec/changes/`.

## Goals / Non-Goals

**Goals:**
- Display all active OpenSpec changes as cards with progress, artifact links, and import action
- Display archived changes in a collapsible section at the bottom
- Extend `watchers.rs` so new change directories appear within 2 seconds without a manual refresh
- Shell `openspec-beads-import <change>` from the UI and surface output in a modal
- Detect "already imported" state from the task list (epic with `openspec:<name>` label)
- Add "Changes" nav entry to the layout shell (≤ 2 clicks from anywhere)

**Non-Goals:**
- Editing or creating OpenSpec artifacts from the UI
- Running `openspec validate` from this view (covered by openspec-spec-panel)
- Real-time streaming of import output via Tauri event channel (blocking `CommandOutput` is simpler and sufficient for a ~5s operation)
- Pagination of the change list (the number of changes is bounded and small)

## Decisions

### D1: Navigation placement

The existing `ViewSwitcher` component (rendered in `TopBar` in `layout/index.tsx`) controls the `view` value in `HashStateContext`. The "Changes" entry is added as a new button in `ViewSwitcher` alongside the existing view options (`list`, `focus`, `ready`).

`TaskListPanel` already uses `state.view` to branch between `FocusView`, `ReadyToStartView`, and the default task list. A new `state.view === 'changes'` branch mounts `ChangesBrowser` in the left panel. The right detail panel remains visible (showing the selected task or empty state), which is consistent with the existing pattern.

**Alternative considered**: Dedicated full-width route for the Changes view — rejected because the existing view-switching pattern (left panel content swap) avoids introducing React Router and is consistent with the current architecture.

### D2: Filesystem watcher for `openspec/changes/`

New `OpenSpecWatcher` struct in `watchers.rs`, following `JsonlWatcher` exactly:

```rust
pub struct OpenSpecWatcher {
    project_path: String,
    app: AppHandle,
}
```

Watches `<project_root>/openspec/changes/` with `RecursiveMode::NonRecursive`. Fires on `EventKind::Create` and `EventKind::Remove` (directory-level events). Debounces with 500ms quiet period, then emits:

```rust
app.emit("changes_list_changed", ChangesListChangedPayload { project: project_path })
```

`WatcherRegistry` stores `(PollHandle, WatchHandle, OpenSpecWatchHandle)` per project. When `connect_project` succeeds, both `JsonlWatcher` and `OpenSpecWatcher` are started. When `stop_project` is called, both are stopped.

**Tauri event name**: `changes_list_changed` — distinct from `task_list_changed` to avoid spurious task list re-fetches.

**Alternative considered**: Single watcher watching the whole project root recursively — rejected because it triggers on every file change in the project, not just changes to the `openspec/changes/` directory structure.

### D3: `list_changes` Tauri command structure

```rust
#[derive(serde::Serialize, specta::Type)]
pub struct ChangeInfo {
    pub name: String,
    pub is_archived: bool,
    pub last_modified: Option<u64>,   // Unix timestamp ms
    pub artifacts_present: ArtifactFlags,
}

#[derive(serde::Serialize, specta::Type)]
pub struct ArtifactFlags {
    pub proposal: bool,
    pub design: bool,
    pub specs: bool,   // true if specs/ subdirectory exists and is non-empty
    pub tasks: bool,
}
```

`list_changes()` reads `AppState::current_project()` for the project root. If no project is connected it returns an empty vec rather than an error, so the frontend can render the empty state without an error boundary.

`last_modified` is the mtime of the change directory itself (most recent of the directory's own mtime and `tasks.md` mtime). Returned as Unix epoch milliseconds so TypeScript can construct a `Date` object directly.

### D4: `get_change_progress` implementation

Reads `<project_root>/openspec/changes/<change>/tasks.md` (or the archived path). Parses line-by-line:
- `- [x]` or `- [X]` → done + 1
- `- [ ]` → total + 1

Returns `ChangeProgress { done: u32, total: u32 }`. If `tasks.md` is absent, returns `{ done: 0, total: 0 }` — the frontend renders "No tasks yet" when `total == 0`.

Both active and archived changes use the same command — the `is_archived` flag on `ChangeInfo` determines the path prefix.

### D5: `import_change_to_beads` — blocking CommandOutput

```rust
#[derive(serde::Serialize, specta::Type)]
pub struct CommandOutput {
    pub stdout: String,
    pub stderr: String,
    pub exit_code: i32,
}
```

Uses `std::process::Command::output()` (blocking, spawned in `tauri::async_runtime::spawn_blocking`). The `openspec-beads-import` skill typically completes in 3–8s, so a blocking call is acceptable. The UI shows a spinner in the modal while the command runs; the modal does not stream — it shows the full output once the future resolves.

Timeout: 30s. If the command times out, the modal shows a "Timed out" error with the partial stdout if any.

**Alternative considered**: Streaming via Tauri event channel using a spawned child process — rejected because it requires a separate event name, a progress listener on the frontend, and a process handle cleanup path, all for a ~5s command. The blocking approach is simpler and the UX difference is negligible.

### D6: "Already imported" detection

The `ChangesBrowser` component reads the already-loaded task list from TanStack Query (`useTasks` hook) and looks for any task with `task_type = 'epic'` and a label matching `openspec:<change-name>`. This is a pure client-side check — no new Tauri command needed.

When found:
- The import button is replaced with "Already imported — view epic"
- Clicking it sets `state.selectedTask = epicId` in `HashStateContext` and `state.view = 'list'`

When not found:
- "Import to beads" button is shown

This detection runs reactively — if the import succeeds and the task list updates (via `task_list_changed` event + DoltPoller), the button state updates automatically within the polling interval.

### D7: Archived section collapse state

Collapse state is stored in component-local React state (`useState(true)` — collapsed by default). This is not persisted to the Tauri store; the user must re-expand it each session. This is acceptable because archived changes are infrequently accessed.

**Alternative considered**: Persist to `layout.json` via Tauri store — rejected for the initial implementation; can be added as a follow-up if users request persistence.

### D8: Artifact links via `shell::open`

`shell::open(path)` opens the file in the system default editor. The full path is constructed as:
```
<project_root>/openspec/changes/<change-name>/<artifact>
```

For the `specs` artifact link, clicking opens the `specs/` directory (which opens in Finder/Explorer) since specs is a directory, not a single file.

The artifact link is greyed out (low opacity, no click handler) when `ArtifactFlags.<artifact>` is false on the `ChangeInfo`.

### D9: ChangesBrowser component structure

```
ChangesBrowser
├── (active changes list)
│   └── ChangeCard × N
│       ├── ProgressBar
│       ├── ArtifactLinks
│       └── ImportButton → ImportModal
└── ArchivedSection (collapsible)
    └── ChangeCard × M (read-only: no ImportButton)
```

`ChangesBrowser` calls `list_changes()` on mount and re-fetches on `changes_list_changed` event (via `listen` from `@tauri-apps/api/event`). Each `ChangeCard` calls `get_change_progress(change.name)` individually so progress bars can load lazily.

`ImportModal` is mounted by `ChangeCard` and conditionally rendered (`isOpen` boolean). It fires `import_change_to_beads(change.name)` when opened (via `useEffect`), shows a spinner, then transitions to success or error state.

## Risks / Trade-offs

- **`openspec-beads-import` not on PATH in Tauri process** → Per usability-initiative design risk, the Tauri sandbox on macOS may not inherit the user's shell PATH. Mitigation: resolve `openspec-beads-import` at startup via `which openspec-beads-import` (or use the same `bd_path` pattern from D4 in the initiative design). If not found, the ImportButton is disabled with a tooltip: "openspec-beads-import not found — run from terminal."
- **`openspec/changes/` directory may not exist** → If the project has no `openspec/changes/` directory, `list_changes()` returns an empty vec and the view shows "No OpenSpec changes found". The watcher gracefully fails to start (no-op), which is the same behavior as `JsonlWatcher` when `.beads/` is absent.
- **Archived directory may not exist** → `list_changes()` checks for the `archive/` subdirectory; if absent, `is_archived` changes are simply absent. No error.
- **`WatcherRegistry` type change** → Adding `OpenSpecWatchHandle` to the stored tuple changes the type signature of `WatcherRegistry`. This is a contained refactor — only `watchers.rs`, `project.rs` (`connect_project` and `disconnect_project`), and any tests are affected.

## Migration Plan

All changes are additive. No existing data model or Tauri command changes. The `OpenSpecWatcher` is started alongside `JsonlWatcher` in `connect_project`; both are stopped in `stop_project`. The layout change (`state.view === 'changes'` branch) is a pure addition. No existing view behavior is modified.
